# Beacon Landing Page Audit
**Date:** 2026-05-04  
**Scope:** Audit Beacon's current homepage against the HireProof homepage structure the user provided, then define how Beacon should market itself more clearly.

---

## Executive Summary

Beacon's current homepage is **technically correct but commercially weak**.

It explains:
- that Beacon has memory
- that Beacon has frameworks
- that Beacon has multiple surfaces
- that Beacon has architecture

But it does **not** explain fast enough:
- what real problem Beacon solves
- who should use it first
- why the graph exists
- what a "good run" actually looks like
- why framework-guided deep research matters in a concrete workflow

HireProof's page is stronger because it is **report-first and risk-first**. It tells the user exactly:
- what to paste
- what risk will be detected
- what verdict comes back
- why the output can be trusted

Beacon should not copy HireProof's topic. But it should copy the same clarity:
- concrete input
- concrete workflow
- concrete output
- concrete reason to trust it

---

## Current Homepage Audit

## What Beacon does well today

- The visual identity is strong and distinct.
- The page clearly signals that Beacon is not just a chatbot.
- The homepage already exposes the main surfaces: trial, dashboard, docs, support, graph, and memory.
- The framework count is visible and differentiated.
- The architecture scene makes the product feel more agent-native and system-level than a normal search UI.

## What the homepage does poorly today

- The hero is too abstract.
  "Research that keeps getting better after the first run" is true, but it is not enough for a first-time visitor to know what Beacon is for.

- The page leads with platform mechanics before use cases.
  Memory, harness, graph, MCP, and access surfaces are explained before the visitor sees a vivid real-world problem to solve.

- The graph is shown before it is justified.
  A graph is not a selling point by itself. It needs a job:
  - show where evidence came from
  - show what changed between runs
  - show how research compounds over time

- The frameworks section is under-marketed.
  It says frameworks change behavior, but it does not tell the visitor when they should care.

- The API section is directionally useful but commercially misleading.
  It makes Beacon look easy to try via HTTP, but the real product path is authenticated and key-dependent.

- The homepage feels like it was written by the builder for technical reviewers.
  It is not yet written for a user who is asking:
  "What problem does this solve better than ChatGPT, Deep Research, or Perplexity?"

---

## Why HireProof's Homepage Works

The HireProof page succeeds because it stays anchored to one user fear:

> "This job post might be fake. Check it before you apply."

That lets the rest of the homepage stay coherent:
- suspicious input
- evidence gathering
- verdict
- risk signals
- training mode
- automation integrations

Everything on the page reinforces one thing:

> "Paste a suspicious work offer and get an evidence-backed verdict."

Beacon needs the same level of clarity, even if the category is broader.

---

## The Best Beacon Positioning

Beacon should not lead with:
- "durable workflows"
- "agent surfaces"
- "MCP"
- "context, memory, harness"

Those are supporting details, not the first sell.

Beacon should lead with:

> "Turn a research question into a reusable evidence base that gets smarter every rerun."

Or more concretely:

> "Use Beacon when one research session is not enough."

That is the real wedge.

Beacon is strongest when the user needs:
- repeated research on the same topic
- visible evidence trails
- structured research methods
- delta tracking over time
- reusable memory instead of one-off chat output

## Best Use Cases To Feature

Beacon should market around **3 to 5 real use cases**, not generic "research."

## 1. Hackathon validation and problem discovery

This is the strongest use case to feature right now because it is authentic to the product and to the project context.

Beacon helps a hackathon builder answer:
- Is this problem real?
- Who already complains about it?
- What alternatives do people use today?
- What changed in the last 30 to 90 days?
- Is the market signal strong enough to build now?

This use case also makes the frameworks matter:
- Jobs To Be Done
- Problem/Solution Fit
- Opportunity Solution Tree
- Market Map
- RICE
- SWOT
- PESTLE

This should be a top homepage section.

Suggested framing:

> "Validate whether the problem is real before you build."

## 2. Competitive research over time

Use Beacon when the same company, market, or category needs to be tracked weekly.

Example:
- "AI coding agents in 2026"
- "Vercel vs Cloudflare developer platform moves"
- "Open-source browser agents"

The pitch:

> "Run once for the baseline. Run again for what changed."

## 3. Product strategy and framework-led research

This is where Beacon is much stronger than generic chat tools.

It should explicitly say:
- same topic, different frameworks, different outputs
- not just search
- not just summary
- structured investigation

Suggested framing:

> "Use deep research frameworks when you need more than a generic answer."

## 4. Team memory for recurring topics

This is where the memory bank and graph become justified.

Example:
- startup market tracking
- recurring customer pain analysis
- ecosystem monitoring
- partner or competitor watchlists

The pitch:

> "Beacon remembers the map so your team does not restart from zero."

## Why The Graph Should Exist

Right now the graph is interesting, but not fully explained.

It should not be marketed as:
- a cool visualization
- a 3D feature
- a dashboard extra

It should be marketed as a **research trust and reuse tool**.

## The graph's real jobs

### 1. Provenance
The graph shows how a final report traces back to:
- query plans
- source URLs
- extracted memory
- prior runs

### 2. Delta visibility
The graph should help the user see:
- what was already known
- what was added this run
- what changed over time

### 3. Knowledge reuse
The graph is the visual answer to:

> "Why is Beacon different from a one-shot report?"

Because the graph makes durable research state visible.

### 4. Team communication
The graph helps explain a topic to other people:
- founders
- hackathon judges
- PMs
- investors
- teammates

Suggested homepage explanation:

> "The graph is not decoration. It shows how evidence, memory, and reports connect, so you can inspect what Beacon learned and what changed between runs."

---

## Why Frameworks Should Be Marketed Harder

The current homepage says frameworks shape the system behavior.
That is good, but still too internal.

It should instead say:

> "Frameworks keep research from collapsing into generic AI summary mode."

This matters because most people have already seen:
- ChatGPT summarize a topic
- Perplexity gather links
- generic AI produce a neat but shallow answer

Beacon's framework story is:
- different frameworks change the search plan
- different frameworks change what counts as evidence
- different frameworks change how the report is structured

For hackathon validation specifically, Beacon should highlight:
- Problem/Solution Fit
- Jobs To Be Done
- Opportunity Solution Tree
- SWOT
- PESTLE
- RICE

Suggested section headline:

> "Choose the research method before you trust the answer."

## Homepage Messaging Problems To Fix

## Problem 1: The hero is too abstract

Current hero:

> "Research that keeps getting better after the first run."

This is elegant but too vague.

Better direction:

> "Validate a problem, track what changes, and keep the evidence."

Or:

> "Turn one research run into a reusable knowledge base."

## Problem 2: No strong use-case ladder

The homepage should show use cases early, ideally right under the hero.

Recommended set:
- Validate hackathon ideas
- Track a market or competitor weekly
- Run structured deep research with frameworks
- Build reusable team memory around a topic

## Problem 3: The page explains surfaces before outcomes

Trial, dashboard, graph, support, docs, and memory bank are useful, but these are product surfaces, not reasons to buy.

The landing page should move in this order:
1. problem
2. use case
3. workflow
4. trust/reason to believe
5. surfaces
6. integrations/docs

## Problem 4: The graph needs a business explanation

Right now the graph reads like a feature tour item.
It should read like:

> "Inspect where the answer came from and what changed over time."

## Problem 5: The page underuses the hackathon angle

For this product, hackathon validation is not a side example. It is an excellent flagship use case because:
- it is founder-shaped
- it is time-sensitive
- it benefits from deep research
- it benefits from frameworks
- it benefits from reruns as the market shifts

## Recommended Homepage Structure

## 1. Hero

Headline:

> "Validate a problem, track what changes, and keep the evidence."

Subhead:

> "Beacon is a durable research agent for questions that need more than one pass. It plans deep web research, applies structured frameworks, stores what it learned, and turns reruns into delta reports instead of starting from zero."

Primary CTA:
- Run sample brief

Secondary CTA:
- See hackathon validation flow

## 2. Use cases

Section title:

> "Built for research that needs a second run."

Cards:
- Validate a hackathon idea
- Track a market weekly
- Compare competitors with frameworks
- Build reusable topic memory

## 3. How Beacon works

Section title:

> "From question to reusable evidence base."

Flow:
1. Start with a research question
2. Pick a framework
3. Beacon plans deep search
4. Report + sources + memory are saved
5. Rerun later and see only what changed

## 4. Why the graph exists

Section title:

> "See how the answer was built."

Points:
- links reports to source evidence
- shows memory over time
- highlights what is new vs already known
- makes the system auditable

## 5. Framework section

Section title:

> "Deep research is better when the method is explicit."

Highlight:
- 47 frameworks
- same topic, different lenses
- hackathon validation frameworks called out by name

## 6. Trust section

Section title:

> "Why Beacon is different from a one-shot AI summary."

Points:
- persistent memory
- delta reports
- source ledger
- visible graph
- recurring runs

## 7. Surfaces and integrations

Only after the above.

Then mention:
- trial
- dashboard
- memory bank
- graph
- API
- MCP
- docs

## Recommended Hackathon-Focused Section

Beacon should add a dedicated section like this:

### Validate a hackathon idea before you build

Use Beacon to check:
- whether the problem appears in the wild
- who feels the pain most
- what alternatives already exist
- how crowded the solution space is
- whether recent market movement suggests urgency

Suggested supporting text:

> "Use Jobs To Be Done, Problem/Solution Fit, SWOT, RICE, and Market Map frameworks to test whether a hackathon idea is solving a real problem or just sounding good in a pitch."

This is one of the most important marketing upgrades Beacon can make.

## Hackathon validation copy

> "Before building, validate the problem. Beacon helps hackathon teams test whether the pain is real, whether users already talk about it, what existing solutions miss, and whether the timing is right."

## Delta copy

> "Run once for the baseline. Run again for what changed."

## Practical Marketing Direction

Beacon should be marketed less like:
- an agent platform
- a research infrastructure stack
- a workflow architecture demo

And more like:
- a durable validation engine
- a framework-guided deep research agent
- a recurring market intelligence system

Best short positioning line:

> "Beacon helps you validate a topic deeply once, then track what changes without restarting from zero."

Best hackathon-specific line:

> "Use Beacon to pressure-test whether your hackathon idea solves a real problem before you spend the weekend building it."

---

## Recommended Next Deliverables

1. Rewrite the homepage hero around problem validation and durable evidence.
2. Add a use-case section led by hackathon validation.
3. Add a graph explanation section so the visualization has a clear job.
4. Rewrite the frameworks block so it explains why method matters.
5. Replace generic platform-first copy with outcome-first copy.
6. Keep the architecture and access-surface details, but move them lower on the page.

---

## Bottom Line

HireProof's homepage works because it tells one clear story:

> suspicious input -> evidence search -> verdict

Beacon needs an equally clear story:

> important question -> framework-guided deep research -> saved evidence -> smarter rerun

The graph, memory, and workflows are good differentiators.
But they become convincing only after the homepage explains the user problem first.
